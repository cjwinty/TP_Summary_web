# Contextual Knowledge Base

## Goal
The TP Summary application helps users analyse support tickets by retrieving structured information from multiple sources: **Targetprocess (TP) API Comments**, **Structured Entity Data**, and **Ticket Summaries**. By integrating these sources, we provide a comprehensive context for LLM summarisation and Q&A.

## Core Concepts

1.  **Entity:** A core record in the system (e.g., Request, Bug, Feature). Entities are managed by their unique `entity_id` and belong to a specific `entity_type`.
2.  **Project:** A high-level grouping for multiple related entities (`project_id`, `project_name`). This helps scope searches and context gathering.
3.  **Metadata/Attributes (Entity Data):** Structured data attached to an entity, including:
    *   `description`: The primary text description of the item.
    *   `entity_state`: Current state (e.g., Open, Closed).
    *   `project_id`/`project_name`: Project affiliation.
    *   `client`, `product`, `release_version`, `site`: Key commercial metadata used for filtering and context injection.
    *   Custom fields stored both as dedicated SQL columns (for queryability) and as raw JSONB (for LLM context and unmapped fields).

## Database Tables

| Table | Purpose |

## Data Sources & Retrieval Methods

### 1. Comments (Source: TP API v2)
Raw comments retrieved via `GET /Comments`.
*   **Schema:** Stored in `comments` table with `request_id`, `comment_data`, `entity_type`, `fetched_at`.
*   **Context Value:** The comment text itself, associated with an Entity ID and Type.
*   **Caching:** `get_comments()` checks local DB first; `use_cache=False` forces a fresh API fetch.

### 2. Summaries (Source: Derived from Comments)
Automated summaries generated by the LLM that condense large volumes of comments into a single, digestible block of text.
*   **Schema:** Stored in `summaries` table.
*   **Context Value:** The summarised text blob.

### 3. Entity Metadata (Source: TP v1 API)
Detailed metadata fetched via the v1 API using the `V2_TO_V1_ENDPOINT` mapping (Request→requests, Bug→bugs, etc.). Uses `xml.etree.ElementTree` for namespace-agnostic XML parsing.
*   **Extracted fields:** `entity_state`, `entity_state_id`, `description`, `create_date`, `project_id`, `project_name`, custom fields.
*   **Custom field columns (9):** `customer_ref`, `internal_priority`, `support_level`, `next_action`, `paid_work`, `downtime`, `out_of_hours`, `customer_chased_date`, `stop_feedback_request`.
*   **Storage:** Both dedicated SQL columns AND raw `custom_fields` JSONB column.
*   **Unsupported entity types** (Period, Timesheet, etc.) are silently skipped via `V2_TO_V1_ENDPOINT` check — no API call is made, logged at DEBUG level.
*   **Relations:** Fetched via v2 `/Relation` API, stored in `entity_relations` table.

### 4. Project Names (Source: TP v2 API)
Fetched via `get_all_projects()` → `GET /Project?take=200` (v2 JSON API). Returns `{"id": ..., "name": ...}` items. Used to backfill human-readable project names into `entity_data.project_name`.

## How Context is Assembled

*   **RAG/Chatbot:** Prioritises vector search against `embeddings` using a combination of comment and summary chunks, filtered by project and client if provided.
*   **Summarisation:** Concatenates all available comments AND the master summary text into one block for LLM digestion, while explicitly mentioning the entity's type and ID in the prompt header.

## Key Variable Injections (For Prompts)
The system automatically injects variables like `{{entity_type}}`, `{{project_name}}`, etc., into prompts to ground the LLM response in specific technical details of the ticket/record being analysed.

## Spelling Convention
All code identifiers, route paths, templates, and database prompt names use UK English (`-ise` not `-ize`): `summarise`, `optimise`, `analyse`, `initialise`, `normalise`.

## Settings: Cache Management

The Settings page provides a unified Cache Management section covering all data tables:

1. **Cache Stats** — Live row counts for Comments, Summaries, Entity Data, Entity Relations, Custom Fields, Embeddings, Chat History
2. **Health Check** — Row counts, orphan detection (entity_data, entity_relations, request_custom_fields, embeddings), index listing, DB size
3. **Optimise DB** — VACUUM ANALYZE + orphan cleanup across all tables
4. **Clear buttons** — Individual clear for Entity Data, Entity Relations, Summaries, Chat History, plus a Clear All Cache (deletes everything in FK-safe order)
5. **Entity Metadata Backfill** — SSE-based backfill that finds entities with comments but no entity_data and fetches+stores metadata
6. **Resolve Project Names** — SSE-based, fetches all projects via single TP API call and backfills NULL `project_name` values
7. **Cache Range** — SSE-based, cache a range of entity IDs with Smart (missing only) or Force (re-fetch all) mode, progress bar

## Settings: Entity Metadata Backfill Behaviour

- **Smart (cache range):** Skips entities that already have comments; refreshes entity_data only for entities with NULL project names; reports separate counts for new fetches vs metadata-only refreshes
- **Force (cache range):** Deletes existing embeddings, re-fetches comments and entity metadata for every entity in range
- **Metadata backfill:** Only processes entities that have comments but no `entity_data` row

## Unsupported Entity Types
Entity types not in the `V2_TO_V1_ENDPOINT` mapping (e.g., Period, Timesheet) are skipped entirely in `refresh_entity_metadata()` — no v1 entity data call, no relations API call, logged at DEBUG level only.
